# logthing

Logthing is a single-binary Go service that receives syslog messages, stores
them locally, and serves both a REST API and a browser UI for inspecting recent
message history.

## Current shape

- Syslog listener built on `gopkg.in/mcuadros/go-syslog.v2`.
- HTTP API and frontend served by the same Go process.
- HTTP Basic Auth for both `/api/v1/*` and the frontend.
- Vite/React frontend embedded into the Go binary with `go:embed`.
- Append-only local storage as daily newline-delimited JSON files.

The first storage implementation uses NDJSON rather than Parquet or Iceberg.
That keeps ingestion durable, append-friendly, and easy to inspect while the
service shape is still small. The storage package boundary is intended to make a
later Parquet compaction or replacement store a contained change.

## Configuration

| Env var | Default | Description |
| --- | --- | --- |
| `LOGTHING_USERNAME` | required | Username for HTTP Basic Auth. |
| `LOGTHING_PASSWORD` | required | Password for HTTP Basic Auth. |
| `LOGTHING_HTTP_ADDR` | `:8080` | HTTP listen address for the REST API and frontend. |
| `LOGTHING_SYSLOG_UDP_ADDR` | `:5514` | UDP syslog listen address. Set to an empty string to disable UDP. |
| `LOGTHING_SYSLOG_TCP_ADDR` | `:5514` | TCP syslog listen address. Set to an empty string to disable TCP. |
| `LOGTHING_SYSLOG_FORMAT` | `automatic` | Parser format: `automatic`, `rfc5424`, `rfc3164`, or `rfc6587`. |
| `LOGTHING_TEST_EVENT_NETWORK` | derived from listener config | Test-event sender network: `udp` or `tcp`. |
| `LOGTHING_TEST_EVENT_TARGET` | derived from listener config | Test-event sender destination, usually `127.0.0.1:5514`. |
| `LOGTHING_DATA_DIR` | `data/messages` | Root directory for local message files. |

Port `514` normally requires elevated privileges. The default syslog port is
therefore `5514` for local development.

## Build and Run

Install frontend dependencies once:

```powershell
pnpm --dir frontend install
```

Build the frontend into the generated embedded asset directory:

```powershell
pnpm --dir frontend build
```

Run tests:

```powershell
make test
```

`make test` intentionally builds the frontend before running Go tests. The Go
server embeds `internal/web/dist`, and that directory is generated by Vite rather
than committed.

Run the service:

```powershell
$env:LOGTHING_USERNAME = "admin"
$env:LOGTHING_PASSWORD = "secret"
go run ./cmd/server
```

Open `http://localhost:8080/` and sign in with the configured username and
password when the browser prompts for credentials.

## Syslog Sender CLI

The `cmd/syslogsend` tool sends a single RFC5424 syslog message to a specified
destination. It is useful for local smoke tests and for testing remote Logthing
instances.

```powershell
go run ./cmd/syslogsend `
  -network udp `
  -addr 127.0.0.1:5514 `
  -message "hello from syslogsend"
```

TCP is also supported:

```powershell
go run ./cmd/syslogsend -network tcp -addr 127.0.0.1:5514 -message "tcp test"
```

Useful flags include `-hostname`, `-app`, `-procid`, `-msgid`, `-facility`,
`-severity`, and `-timeout`.

## REST API

Interactive API documentation is served from the same binary at
`/swagger-ui/`. The raw OpenAPI document is available at `/swagger.json`.
Both endpoints use the same Basic Auth credentials as the application UI.

Utility endpoints:

| Method | Path | Description |
| --- | --- | --- |
| `GET` | `/healthcheck` | Basic service health response. Does not require authentication. |
| `GET` | `/swagger-ui/` | Embedded Swagger UI. Requires Basic Auth. |
| `GET` | `/swagger.json` | OpenAPI document. Requires Basic Auth. |

Authenticated:

| Method | Path | Query | Description |
| --- | --- | --- | --- |
| `GET` | `/api/v1/messages` | `q`, `host`, `limit`, `offset`, `since`, `until` | Returns latest-first messages from local storage. |
| `GET` | `/api/v1/messages/stream` | none | Streams newly appended messages as Server-Sent Events. |
| `POST` | `/api/v1/messages/import` | none | Imports newline-delimited JSON messages into local storage. |
| `POST` | `/api/v1/test-event` | none | Sends one server-side RFC5424 test event to the configured syslog destination. |

`host` is an exact syslog hostname filter and can be repeated for multiple
hosts. `since` and `until` use RFC3339 timestamps and filter on `received_at`.
`limit` defaults to `200` and is capped at `2000`. `offset` defaults to `0` and
is used for paging.

Example:

```powershell
curl.exe -u admin:secret "http://localhost:8080/api/v1/messages?limit=50&offset=0&host=edge-1&q=error"
```

Stream new messages with SSE:

```powershell
curl.exe -N -u admin:secret "http://localhost:8080/api/v1/messages/stream"
```

The response shape is:

```json
{
  "data": [
    {
      "id": "f2e4a19c6f50429bbf7afc1fcb705fa6",
      "received_at": "2026-06-19T12:00:00Z",
      "timestamp": "2026-06-19T12:00:00Z",
      "transport": "udp/tcp",
      "source": "127.0.0.1:51234",
      "priority": 13,
      "facility": 1,
      "severity": 5,
      "hostname": "edge-1",
      "app_name": "router",
      "proc_id": "123",
      "msg_id": "abc",
      "tag": "router",
      "message": "link changed"
    }
  ],
  "meta": {
    "count": 1,
    "limit": 50,
    "offset": 0,
    "has_more": false
  }
}
```

Import accepts NDJSON in the body. Blank lines are skipped, and invalid JSON,
timestamps, facilities, or severities return `400` with the failing line number.
Imported messages use the imported timestamp for both `timestamp` and
`received_at`, so they sort into the historical timeline. Supported fields are
`timestamp`, `host`, `program`, `pid`, `severity`, `facility`, and `message`.
Facility and severity may be syslog names such as `daemon` and `warning` or
their numeric values.

Example:

```powershell
curl.exe -u admin:secret `
  -H "Content-Type: application/x-ndjson" `
  --data-binary "@router.ndjson" `
  "http://localhost:8080/api/v1/messages/import"
```

## Frontend Workflow

The UI is the first screen at `/`. It shows the message timeline in a wide,
scrollable table with syslog fields as columns. The host selector maps to
repeated `host` query parameters, the filter box maps to `q`, and the page
controls map to `offset`. Live updates are available only on page 1 and use
`/api/v1/messages/stream` so newly received messages are pushed into the visible
window. If the SSE connection fails, live updates are disabled and the UI shows
a notification. The import button uploads NDJSON to
`/api/v1/messages/import`, and the export button downloads the currently visible
rows as NDJSON. The `Send test event` button calls `/api/v1/test-event`, which
asks the server to send one syslog message to `LOGTHING_TEST_EVENT_TARGET`.

Browsers cannot send raw UDP or TCP syslog packets directly through normal web
APIs, so the button uses a server-side sender. The CLI sender above is the
direct way to send a syslog packet from a terminal.

## Docker

Build the image locally:

```powershell
docker build -t timmersuk/logthing:local .
```

Run the container directly:

```powershell
New-Item -ItemType Directory -Force .\logthing-data | Out-Null
docker run --rm `
  -p 8080:8080 `
  -p 5514:5514/tcp `
  -p 5514:5514/udp `
  --mount "type=bind,source=${PWD}\logthing-data,target=/data" `
  -e LOGTHING_USERNAME=admin `
  -e LOGTHING_PASSWORD=change-me `
  timmersuk/logthing:local
```

Open `http://localhost:8080/` for the UI or
`http://localhost:8080/swagger-ui/` for Swagger UI. Send syslog messages to
TCP or UDP port `5514`.

The image runs as an unprivileged user and stores messages under
`/data/messages`. The command above bind-mounts `.\logthing-data` so the local
NDJSON files are visible on the host and survive container replacement.

## Docker Compose

The included `docker-compose.yml` mirrors the simple run style used by small
log-viewer images: one service, published UI/API port, published syslog ports,
and a bind-mounted local storage folder.

Start it with local defaults:

```powershell
New-Item -ItemType Directory -Force .\logthing-data | Out-Null
$env:LOGTHING_PASSWORD = "change-me"
docker compose up --build
```

Useful compose variables:

| Variable | Default | Description |
| --- | --- | --- |
| `LOGTHING_IMAGE` | `logthing:local` | Image name used by compose. |
| `LOGTHING_USERNAME` | `admin` | Basic Auth username. |
| `LOGTHING_PASSWORD` | `secret` | Basic Auth password. Override this. |
| `LOGTHING_HTTP_PORT` | `8080` | Host port for the UI/API. |
| `LOGTHING_SYSLOG_PORT` | `5514` | Host TCP and UDP syslog port. |
| `LOGTHING_SYSLOG_FORMAT` | `automatic` | Syslog parser format. |
| `LOGTHING_TEST_EVENT_NETWORK` | `udp` | Network used by the UI test-event sender. |
| `LOGTHING_TEST_EVENT_TARGET` | `127.0.0.1:5514` | Destination used by the UI test-event sender inside the container. |
| `LOGTHING_DATA_PATH` | `./logthing-data` | Host folder bind-mounted to `/data`. |

Stop it with:

```powershell
docker compose down
```

## Makefile

The Makefile wraps the common local and Docker commands:

```powershell
make test
make build
make syslogsend
make docker-build IMAGE=timmersuk/logthing TAG=0.1.0
make docker-run IMAGE=timmersuk/logthing TAG=0.1.0
make docker-run IMAGE=timmersuk/logthing TAG=0.1.0 DATA_DIR=C:/logs/logthing
make docker-login
make docker-push IMAGE=timmersuk/logthing TAG=0.1.0
```

For a multi-architecture Docker Hub publish with Buildx:

```powershell
make docker-buildx-push IMAGE=timmersuk/logthing TAG=0.1.0
```

Set `IMAGE` to your Docker Hub namespace, for example
`<dockerhub-user>/logthing`.

## GitHub Actions Releases

Public GitHub repositories can use standard GitHub-hosted Actions runners
without paid minutes. The included `.github/workflows/ci.yml` runs frontend
builds and Go tests on pull requests and pushes, then builds the Docker image.

Docker Hub publishing is enabled for:

- Git tags matching `v*.*.*`, such as `v0.1.0`.
- Published GitHub releases.
- Manual workflow runs with `push_image=true`.

Before pushing images, configure these repository settings in GitHub:

| Setting | Type | Example |
| --- | --- | --- |
| `DOCKERHUB_USERNAME` | Actions secret | `timmersuk` |
| `DOCKERHUB_TOKEN` | Actions secret | Docker Hub access token |
| `DOCKERHUB_IMAGE` | Actions variable | `timmersuk/logthing` |

The workflow publishes semver tags such as `0.1.0`, `0.1`, and `latest` for
tagged releases. Pull requests and normal branch pushes build the Docker image
but do not push it.

## Local Storage

Messages are written as one JSON object per line under the configured data
directory. New writes are split by UTC date and sender source:

```text
data/messages/YYYY/MM/DD/<source>.ndjson
```

For Docker runs, the in-container path is `/data/messages`, which maps to the
host folder configured by `--mount` or `LOGTHING_DATA_PATH`.

The source filename is derived from the `source` field reported by go-syslog.
When that value is `host:port`, the port is stripped so transient UDP source
ports do not create one file per datagram. Characters that are unsafe for file
paths are replaced with `_`, and messages without a source use `unknown`.

Query reads scan all matching source files and return messages in latest-first
order. This is intended for local history and operational browsing, not as a
high-cardinality analytics engine.
